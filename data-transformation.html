Intro

Data transformation is crucial in an Integrated Development Environment (IDE) because it enables developers to manipulate and reshape data to fit the specific requirements of different applications, systems, or data formats. Without effective data transformation capabilities directly within the IDE, the development process would be significantly hindered.

To be more specific, they were used in: Integration and Interoperability, Data Normalization and Standardization, Business Logic Implementation, Performance Optimization, Simplified Debugging and Testing, Code Reusability and Modularity, Rapid Prototyping and Development

DataWeave is a powerful, functional programming language designed by MuleSoft specifically for accessing and transforming data within Mule applications. It's MuleSoft's primary language for handling data manipulation, from simple reformatting to complex mappings between various data structures

02 Research Goal

While DataWeave is an incredibly powerful language, creating the correct DataWeave transformation can be challenging for developer users, especially if they're new to functional programming paradigms. We constantly found customers complaining about transforming data in Mule projects. So I hosted a series of user research activities to collect data and analyze the facts, problems in Anypoint Studio and opportunities in Anypoint Code Builder.

The ultimate goal of the research is to identify:

  • Data Transformation use, frustration and potential resolutions;
  • DataWeave Documentation explorations.

03 Understand Users

Surveys and follow-up interviews with our user "ambassadors" revealed two key user groups and their unique behaviors and preferences.

Expert

More experienced users

The expert users are daily users of the language that usually possess extensive coding background.

  • Starts with Documentation;
  • Learns by writing code and seeing how it operates;
  • More expert level examples;
  • Likes to solve problems themselves.

Novice

Users that are just starting

Novice users are those that have been recently assigned (within a year) and are now starting to use the Dataweave language to do data transformation for designing integrations.

  • Training and Google searches;
  • Learns by tinkering with light version to learn syntax and style;
  • Detailed examples;
  • Relies on help from colleagues and MuleSoft support teams.

04 Identify Problems

Through eight 45-minute sessions with user community members and three 90-minute sessions with senior solution engineers, I gathered qualitative data. This allowed me to identify key insights by grouping similar feedback clusters.

Complexity in DataWeave Language

Developer mental model mismatch (for expert users)

Dataweave has a Developer mental model mismatch. Many developers, especially those that have been more recently taught to code, come to Dataweave with coding background in an Object Oriented style.

"Dataweave, being a functional language causes confusion and frustration as devs go about coding in dataweave as if it were an OO language. Devs need to erase their previous knowledge to be able to come at Dataweave with a âblank slateâ and learn Dataweaveâs style.Â "

Concise but complex syntax

DataWeave's syntax is very concise, which is great for experienced users but can be daunting for newcomers. Mastering the various operators ($ for value, $$for key,$$$ for index, .* for descendants, . for direct access, etc.) and understanding their nuances is crucial and takes practice. Misunderstanding these can lead to unexpected results.

"To do anything more complex requires using DataWeave. I have to crack open google to look up some esoteric function for DataWeave. Why does this require me to learn a new language - It is a stumbling block."

Missing/Insufficient Features in Development Tool

Lack of onboarding experience (for novice users)

As a specialized, functional programming language, DataWeave's distinct syntax and concepts often lead to a significant learning curve. Unfortunately, there's currently no dedicated onboarding experience for DataWeave transformations within Anypoint Studio.

Data visibility

Effectively transforming complex data often requires a good understanding of the input and output schemas, which can be challenging when dealing with diverse systems. Without seeing the data, effectively transforming complex data becomes a significant challenge, often leaving users feeling like they're working with a "black box."

Unreliable validation and testing tool

When writing a script the system will try to validate and will produce an error when the user is not really expecting to see one. The error causes doubt that user is going about creating a script correctly.
Error messages can sometimes be vague or not immediately point to the root cause, especially for type mismatches or structural issues. This can make troubleshooting frustrating.

"When you create a transformation- when trying to save it will still produce an error. If iâm a user, iâll think something is wrong with my script. The project may be working but the error is still there. Refreshing the project helps."

Poor Performance

Long processing time

Resolving metadata and processing DataWeave script can significantly slow down the user's ability to create and test data transformations in an Integrated Development Environment. This delay often stalls the iterative development process, making it frustrating to quickly build and validate transformation logic.

"It takes so long in Studio, so I always open the playground to test my code."
"Large scripts with many uses at once can use a lot memory which can cause some APIâs to take 5 seconds or longer - which is unacceptable."

Workspace Corruption

Over time, the workspace metadata within the Integrated Development Environment (IDE) can become cluttered with outdated or conflicting references. This "untidy" state often leads to unexplainable conflicts, cryptic errors, or general instability, forcing users to frequently resort to restarting the IDE or even performing a full reinstallation to restore functionality.

05 Opportunities

I analyzed the data and hosted a workshop to present the findings and discuss the opportunities that exist in experience design with the researchers, other product designers, product managers and DataWeave team.

Auto-convert or 'Quickfix'

When the desired output data model is already defined, the system can propose transforming the user's input into the correct data structure. This eliminates the need for users to recall specific functions or write error-free code, while also significantly saving time.

Visualization

For expert users +Â novice users

Visualizing metadata structures and data transformations allows users to better grasp both the source and target data models, as well as the transformation process itself. This approach significantly reduces the learning curve for new users and boosts the productivity of experienced users.

Real-Time Preview

For expert users +Â novice users

Seeing the immediate output of transformations empowers users to rapidly iterate on code, significantly accelerating development time. This instant feedback also serves as a powerful learning tool, helping users grasp concepts and refine their skills much faster.

UI-Driven Transformations

For novice users

Interacting with a user interface to establish data transformations from input to output will be a game changer for novice users. Observing how the system generates DataWeave code based on their UI input also provides a highly effective and intuitive learning experience.

System Feedback Enahncement

For expert users +Â novice users

Even if the system experiences some latency, especially in the near future, clear UI indicators will significantly alleviate user anxiety and confusion. Furthermore, accurate and meaningful error messages will empower users to resolve issues independently.

Our users have expressed significant interest in leveraging generative AI for transformations, highlighting two key applications: predicting data mappings based on input and output metadata, and generating transformations directly from user prompts.
